Updated how contour maps are rendered - 2025/09/06

Before, when the contour map was rendered per level, it generated by drawing a line from one point to the next randomly. By repeating this process it managed to create a contour map. However, if I want to make the lines into curves and other shapes, I need to determine what shapes each line forms. I've implemented this by building an adjacency map for each edge, then traversing it until it reached the end (then traversing it again in the opposite direction until I hit the other edge) for lines that hit the border of the rendering container, or until it loops again back to the start.

Testing Contour Map for Background - 2025/09/02

To design a presentable homepage, I've decided to make a dynamic contour map as the background. I created the background by making an array of points which is then applied to a Perlin noise generated random background. Then a Marching Square Algorithm by the The Coding Train is used to generate the lines. The points are then seperated into different groups based on the layer. The contour map is then animated by changing the z value of the Perlin noise over time.
Some features I would like to implement in the future are:

  • Apply some kind of rendering like Bezier curves to curved portion of the contour map so that it doesn't look so jagged
  • Sliding scales and a way to change the values easily. Right now it is a pain to make minor adjustments
  • Way to change the FPS of the animation - Some of my friends who I've asked for feedback says that it needs a flash warning, while others say that it is too jagged and needs to run smoother
  • Related to the previous point, a way to make it more efficient. Currently if I run it too fast, it consumes all my CPU resources and RAM.

Design sprint - 2025/01/17

I was reading about this concept of a design sprint - A way to design and test a prototype in 5 days. This consists of a step in 5 days - Understand (the customer), Ideate, Decide (the idea), Prototype, and Test. According to Google Ventures, this design sprint is usually done by a team with a specific goal in mind, but since I'm doing this alone, and planning on updating the website as a whole, I'll be modifying it into:

  1. 01/17 (Friday) - Research the audience and design a prototype
  2. 01/18 (Saturday) - Turn the concepts into more concrete solutions, into basic designs
  3. 01/19 (Sunday) - Create individual components
  4. 01/20 (Monday) - Combine each of the modules into a cohesive website
  5. 01/21 (Tuesday) - Complete it, and get reviews from others (then somehow implement feedback)
